### 1\. **Principios SOLID**

Es un conjunto de cinco reglas de diseño orientado a objetos (Responsabilidad Única, Abierto/Cerrado, Sustitución de Liskov, Segregación de Interfaz e Inversión de Dependencias). Su objetivo es crear software que sea fácil de mantener, escalar y entender a lo largo del tiempo.

### 2\. **Principio de separación de preocupaciones (SoC)**

Este principio establece que el código debe organizarse de manera que cada módulo se encargue de una única responsabilidad bien definida. Esto facilita la comprensión del código y su mantenimiento, ya que los módulos son más independientes y menos propensos a errores.

### 3\. **Principio de no repetición (DRY)**

Este principio indica que la información debe almacenarse en un único lugar y no se debe duplicar en diferentes partes del código. Esto evita redundancias, reduce la posibilidad de errores y facilita la actualización del código.

### 4\. **Principio de menor abstracción posible (KISS)**

Este principio recomienda utilizar el nivel de abstracción adecuado para cada problema. Un exceso de abstracción puede dificultar la comprensión del código, mientras que una falta de abstracción puede hacerlo rígido e inflexible.

### 5\. **Principio de diseño por contrato (DbC)**

Este principio establece que las interfaces de las clases deben definir contratos que expliquen las condiciones previas, las condiciones posteriores y las invariantes de cada método. Esto facilita la detección de errores y mejora la comunicación entre los desarrolladores.

### 6\. **Patrones de diseño**

Los patrones de diseño son soluciones reutilizables a problemas comunes de diseño de software. Existen muchos patrones de diseño diferentes, cada uno con su propio propósito y ámbito de aplicación. Utilizar patrones de diseño de forma adecuada puede mejorar la calidad, la flexibilidad y la mantenibilidad del código.

### 7\. **Pruebas unitarias**

Las pruebas unitarias son pruebas automatizadas que se escriben para verificar el funcionamiento correcto de cada unidad de código (clases, funciones, etc.). Escribir y ejecutar pruebas unitarias de forma regular te ayuda a detectar errores tempranamente, mejorar la calidad del código y refactoring con mayor confianza.

### 8\. **Integración continua (CI)**

La integración continua es una práctica de desarrollo de software en la que los desarrolladores integran sus cambios en el código de forma regular y automatizada. Esto permite detectar errores de forma temprana y asegurar que el código siempre se encuentra en un estado compilable y ejecutable.

### 9\. **Entrega continua (CD)**

La entrega continua es una extensión de la integración continua que automatiza el proceso de entrega de software a los usuarios. Esto permite entregar nuevas funcionalidades a los usuarios de forma rápida y segura, sin necesidad de intervención manual.

### 10\. **Programación extrema (XP)**

La programación extrema es una metodología de desarrollo de software ágil que se basa en valores como la colaboración, la comunicación y la retroalimentación continua. XP se caracteriza por prácticas como la codificación en pareja, las revisiones de código diarias y las pruebas de aceptación guiadas por el cliente.

### 11\. **Desarrollo impulsado por pruebas (TDD)**

El desarrollo impulsado por pruebas es una metodología de desarrollo de software en la que las pruebas se escriben antes de escribir el código real. Esto ayuda a asegurar que el código cumple con los requisitos y que funcione correctamente.

Estos son solo algunos ejemplos de los muchos principios y directrices que existen para el desarrollo de software. Es importante conocer y aplicar estos principios de forma adecuada para crear código de alta calidad que sea fácil de mantener, escalable y robusto.
Son cinco reglas de diseño de software orientadas a objetos que sirven para que el código sea fácil de mantener, escalar y, sobre todo, para que no se rompa todo cuando tocás una sola línea.
